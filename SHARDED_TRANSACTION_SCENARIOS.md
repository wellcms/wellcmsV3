# 分片事务（Cross-Shard Transaction）适用场景定义

> 本文档用于界定 WellCMS 3.0 及类似论坛/社交系统在什么条件下必须引入分片事务或分布式事务补偿机制。

---

## 一、前提条件

必须**同时满足**以下两点，才构成真实的分片事务需求：

### 1. 单节点物理瓶颈已无法回避
- **存储瓶颈**：单表数据量超过单库承载极限（如 MySQL 单表超过 5000 万~1 亿行，或 PostgreSQL 单库超过 2-4TB）。
- **性能瓶颈**：单节点 TPS/QPS 触顶（如写入超过 2000-5000 TPS），垂直扩容（升级硬件）成本高于水平扩容（加机器）。
- **地理分布需求**：数据必须按地域存储在不同机房（如欧盟用户数据必须在法兰克福节点）。

### 2. 业务存在不可拆分的跨节点原子操作
如果所有相关数据可以通过**同一分片键**路由到同一物理节点，则**不需要**分片事务；只有当业务上必须同时修改两个**不同分片键**对应的数据时，才产生分片事务需求。

## 二、论坛/社交系统中的典型场景

### 场景 1：按用户分片，但发帖需要同步修改版块统计
- **分片策略**：`forum_thread` 按 `user_id` 分片（用户 A 的数据在 shard_0，用户 B 在 shard_1）。
- **跨片事务触发**：用户 A 发帖到「版块 ID = 100」，版块统计表 `forum_category` 在 shard_2（版块表作为全局元数据独立分片或中心库）。
- **原子需求**：帖子写入成功 但 版块 `thread_count++` 失败，会导致版块统计与真实数据不一致。
- **结论**：需要分片事务或 Saga 补偿机制。

### 场景 2：按版块分片，但精华/置顶需要修改用户全局荣誉
- **分片策略**：`forum_thread` 按 `category_id` 分片（不同版块在不同节点）。
- **跨片事务触发**：版主在 shard_0 将帖子加精，同时需要修改 `user` 表（位于 shard_global）的 `forum_feature` 字段。
- **原子需求**：帖子已加精但用户荣誉未增加，或反之，都会引发数据不一致和用户投诉。
- **结论**：需要分片事务或 Saga 补偿机制。

### 场景 3：跨版块移动主题
- **分片策略**：版块 A 的数据在 shard_0，版块 B 的数据在 shard_1。
- **跨片事务触发**：将 thread_id = 123 从版块 A 移动到版块 B，需要同时在 shard_0 删除索引、在 shard_1 插入索引，并扣减/增加两个版块的统计。
- **原子需求**：移动 halfway 会造成"帖子丢失"或"双版块同时存在"。
- **结论**：这是**最典型的必须分片事务场景**。

### 场景 4：资金/积分变动（强一致性场景）
- 用户打赏帖子，涉及 `forum_thread` 的赏金字段、`user` 的余额字段、`forum_message` 的消息记录分别落在不同分片。
- **原子需求**：余额已扣但帖子赏金未加，属于资损事故。
- **结论**：必须强一致，需要 2PC 或 Saga 补偿 + 对账机制。

## 三、什么时候"看起来要分片事务"，但实际上不需要？

### 规避方案：同分片键设计（Co-location）
只要把所有需要原子操作的表**绑定到同一分片键**，就可以完全避免分片事务：

| 反例（需要分片事务） | 正例（不需要分片事务） |
|---|---|
| `forum_thread` 按 `user_id` 分片，`forum_category` 在全局库 | `forum_thread`、`forum_reply`、`forum_category_stats` **全部按 `category_id` 分片** |
| `user` 按 `id` 分片，`forum_thread` 按 `id` 分片 | 用户相关的帖子、回复、统计全部按 `user_id` 分片，版块元数据做冗余副本 |

**工程原则**：优先用**数据模型设计**消除跨片事务需求，而不是引入分布式事务框架。

## 四、决策树

```
是否需要分片事务？
├── 单库是否还能通过垂直扩容满足性能/存储需求？
│   └── 是 → 不需要分片，更不需要分片事务
│   └── 否 → 继续判断
└── 业务涉及的多张表是否可以通过同一分片键路由到同一节点？
    ├── 是 → 设计为同分片键部署，不需要分片事务
    └── 否 → 确认是否存在不可拆分的跨节点原子操作
        ├── 是 → 必须引入分片事务（2PC / Saga / 本地消息表补偿）
        └── 否 → 允许最终一致，可用异步队列替代
```

## 五、WellCMS 3.0 当前状态

- 当前 `config/Database.php` 中 `sharding.tables` **完全为空**，所有表共享同一物理库（default shard）。
- 在此状态下，任何 `Cross-shard query violation` 异常均为**框架路由层误报**，应在 `PdoDriver.resolveShard` 层修复，而不是引入分布式事务。
- 如果未来启用分片，应优先采用**业务域同分片键**策略（如按 `category_id` 或 `user_id` 整体分片），使 `forum_thread` 与其关联统计表落在同一节点，从而继续避免分片事务。
